# Протокол пиксель-приёмки (SM-1)

## Цель

Подтвердить **пиксель-в-пиксель** соответствие каждой из 6 страниц эталону Handoff — это приоритет №1 проекта (SM-1). **Дизайн (отрисованный прототип) — источник истины (AD-13):** матчим визуал по каждому целевому брейкпоинту против отрисованного прототипа, а не DOM. Кросс-страничные расхождения не «примиряем», известные дефекты дизайна воспроизводим дословно, а не «чиним».

Этот каталог — формальный, повторяемый контракт приёмки: README-протокол + по одному файлу-чеклисту на страницу с явным перечнем целевых брейкпоинтов. Он снимает Major-риск «готово определяется на глаз» и служит объективным критерием приёмки страниц Epic 3–6.

## Preview

Канонический способ поднять прод-близкую сборку для приёмки:

```bash
npm run preview
```

Это `next build && next start` — production-build + локальный старт на `http://localhost:3000` (детерминизм — часть пиксель-фиделити, NFR-3). Открываются 6 роутов:

- `/` — Home
- `/about` — About Us
- `/catalog` — Catalog
- `/brands` — Our Brands
- `/shops` — Our Shops
- `/contact` — Contact

Ревьюер открывает роут и **выставляет каждую целевую ширину через responsive-режим DevTools** (device toolbar), сверяя визуал с соответствующим файлом прототипа на каждом брейкпоинте.

`npm run dev` допустим для быстрой итерации, но **отметку `[x]` (приёмку) ставить ТОЛЬКО на `npm run preview`** — dev-сборка (без минификации, иная оптимизация `next/image`) не пиксель-эквивалентна проду.

Если сборка падает — приёмку не начинать, сперва починить сборку. Если порт 3000 занят — `npx next start -p <порт>`.

> **[ASSUMPTION]** Конкретное staging/preview-окружение на инфре Rollun уточняет владелец (Deferred). До этого достаточно локального `npm run preview`.

### Условия ревью (для воспроизводимости)

Пиксель-сверка выполняется человеком, поэтому фиксируем окружение:

- Браузер на движке Chromium, **zoom 100%**, **device-pixel-ratio 1**, окно/DevTools device-mode; расширения, меняющие рендер, отключить.
- Ширину бэнда выставлять **чуть внутри диапазона, а не ровно на границе** (768/820/980/1100…): ширина окна с учётом скроллбара (~15px) может переключить `@media` на самой границе и дать нестабильный результат.

## Методология брейкпоинтов

**Desktop-бэнды** каждой страницы взяты **дословно из `@media`-границ её файла прототипа** (`<Page>.html`) — только границы **≥768px и вплоть до 1280**. Каждый бэнд — отдельный пункт приёмки «визуал совпал с отрисованным прототипом»; в скобках указана представительная ширина для теста.

- **Верхние границы бэндов** — дословно из `@media` прототипа.
- **Нижний порог desktop = 768** — это порог AD-3 (desktop/mobile split), а **НЕ `@media`-граница прототипа**. В desktop-файлах прототипа переключение происходит на 720/560/760, но в прод **ниже 768 отгружается mobile-композиция** (NFR-2/AD-3), поэтому desktop тестируем только до 768. Границы <768 (720/760/560) — mobile-территория, в desktop-чеклист НЕ попадают.

**Порог 768 — владелец один (AD-3):** при `≥768px` рендерится desktop-композиция ⇒ desktop-бэнды **включают** 768 (нижний бэнд `768–…`); mobile-пункт — `<768px` (НЕ `≤768`, иначе пиксель 768 был бы двойным).

**Mobile-композиция** каждой страницы — ровно **один пункт**: `<768px`, letterbox-шелл **440px**, референс `<Page> Mobile.html`. Тестировать на **~390px** (ниже 440-шелла — full-width) И у верхней границы **~760px** (внутри летербокса 440px — иной рендер, чем full-width <440).

## Реестр воспроизводимых дефектов (AD-13 / addendum §I)

Это дефекты **дизайна**, которые воспроизводятся как есть (AD-13: «дизайн — источник истины»). На соответствующем брейкпоинте они помечаются «ожидаемо» и **не заводятся как баги**. Привязка каждого дефекта к странице проверена по прототипу.

**Перечень НЕИСЧЕРПЫВАЮЩИЙ** (AD-13: «включая, но не только §I»). Страница без пункта §I пишет «нет ИЗВЕСТНЫХ постраничных дефектов», а не «дефектов нет».

- **Home (mobile)** — «marketplaces» теряют рейтинги, которые есть на desktop-карточках.
- **About (mobile)** — НЕ содержит desktop-секций контакт-модалки и карты (в mobile-DOM карта отсутствует; desktop-About имеет `.map-section` + модалку). (Cert-lightbox тут НЕТ — он на Our Brands.)
- **Our Brands** — **cert-lightbox** сертификата присутствует только в mobile-DOM (`Our Brands Mobile.html`; `About Us*.html` его НЕ содержат). Esc: на desktop обработчик навешан, но элемент мёртв; на mobile закрытие — только клик.
- **Our Shops** — видимый адрес **Houston TX 77039**, но «GET DIRECTIONS» ведёт на `q=Conroe, Texas`.
- **Contact** — стартовый `src` карты жёстко закодирован `q=53%2F27…Aldine Mail Route Rd, Houston, TX 77039` — это адрес **Houston** (`53/27` — номер дома), воспроизводим как есть. В `Contact.html` слова «Conroe» НЕТ (Conroe — дефект ТОЛЬКО Our Shops).
- **Расхождения таблицы часов между страницами** — это **разные атомы** (AD-14, `hours.store` ≠ `hours.homeCta`), не одно поле, отрисованное по-разному; не примиряем. Помечается только на страницах с таблицей часов (Home, Shops, Contact).

## Правило Done (Epic 3–6)

- Страница Epic 3–6 **не считается Done, пока все её брейкпоинт-пункты (`- [ ]`) не отмечены `[x]`**.
- Каждая эпик-страница ссылается на свой файл-чеклист `checklist-<page>.md`.
- Пункт остаётся `[ ]`, если визуал реально не совпал с прототипом на бэнде (страница не Done). Если расхождение — воспроизводимый дефект из реестра §I, пункт засчитывается совпавшим, а дефект подтверждается в разделе «Ожидаемо».

## Deferred

Авто visual-regression / pixel-diff / CI-гейт — **вне scope, Deferred, не блокирует** приёмку. Ручная брейкпоинт-приёмка по этому протоколу достаточна.
