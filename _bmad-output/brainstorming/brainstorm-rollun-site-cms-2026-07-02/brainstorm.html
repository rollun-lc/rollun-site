03

SCAMPER

семь ходов трансформации решения
Autonomous · коуч
decision · коуч Выбор CMS: Payload CMS 3.0. TS-native, работает ВНУТРИ того же Next.js App Router (нет сетевого хопа, local API), self-host на инфре rollun, авто-генерит админку (менеджер не ломает вёрстку), встроенная Media-коллекция с авто-ресайзом/оптимизацией, access control, open-source. Покрывает все найденные сорта контента.
S Substitute

Вместо свободного page-builder'а — типизированные слоты. Layout захардкожен в React, CMS заполняет только размеченные поля. Менеджер физически не сломает дизайн.

C Combine

Телефон + адрес + соцссылки + часы = один Global «SiteSettings» (паспорт). Правка в одном месте → обновилось в шапке, футере, Contact, Our Shops.

A Adapt

Каждая страница = отдельный Payload Global с полями ровно под её слоты, а не абстрактная «Page». Проще менеджеру, невозможно сломать структуру.

M Modify

Products — коллекция с полем-швом sku/externalId уже сейчас (ручной ввод), чтобы автосинк дописывал цену/картинку/наличие поверх без переделки.

P Put to use

Media-коллекция = авто-оптимизация. Менеджер льёт любую тяжёлую картинку, Payload + next/image режут в нужные размеры/webp. Убирает страх «hero на 8 МБ».

E Eliminate

Юр./статика, вёрстка, микрокопи — НЕ в CMS, гвоздями в коде. В CMS только «живые» слоты. Меньше поверхности для ошибок, быстрее MVP.

R Reverse

Сначала строим пиксель-в-пиксель с захардкоженным контентом (приоритет №1 — дизайн), CMS подключаем слотами ПОТОМ. Дизайн-релиз не блокируется админкой.

decision · коучДизайн-фиделити (приоритет №1): DS-токены из colors_and_type.css → 1:1 в CSS-переменные / Tailwind theme; шрифты Archivo / Hanken Grotesk / Spline Sans Mono; каждый HTML-прототип пересобирается в React пиксель-в-пиксель (матчим визуал, не копируем DOM). Mobile-HTML → адаптив-брейкпоинты.
decision · коучСтек: Next.js App Router + TypeScript + React Server Components тянут данные из Payload local API (тот же процесс). next/image. On-demand revalidation / ISR — правки менеджера появляются быстро без пересборки.

Backcasting

от идеального финала — шагами назад к сегодня
Autonomous · коуч
Идеальная финальная картина

Сайт 1:1 по дизайну в проде. Менеджер меняет картинки / товары / паспорт-контакты через админку Payload, картинки авто-оптимизируются, товары вводятся руками — но модель уже готова к автосинку.

Опорная идея

Уровни текучести контента

Каждый кусок сайта живёт на своём слое — от «меняем ежедневно» до «прибито гвоздями». Слой решает, где контент лежит.

🔴
Живое
картинки, товары — меняются часто
→ CMS · Media + Products
🟡
Правимое
тексты блоков — правятся редко
→ CMS · слоты страниц
🟢
Паспорт
телефон, адрес, ссылки — один источник правды
→ Global SiteSettings
⚫
Гвоздями
вёрстка, структура, юр-текст — только код
→ React · захардкожено
чем выше слой — тем чаще меняется · чем ниже — тем ближе к коду и невозможнее сломать
Модель контента

Globals против Collections

Globals — единичные сущности (паспорт, страницы). Collections — списки записей (товары, бренды, магазины). Users — встроенный auth Payload для менеджеров.

Backcasting · дорожная карта

Фазы 0 → 4

Дизайн релизится первым, CMS наслаивается потом. Ключ: Фаза 1 не ждёт Фаз 2–3.

0

Скаффолд

Next.js + Payload в одном проекте, порт DS-токенов и шрифтов, базовый layout (шапка / футер).

1

Дизайн 1:1 Главная ценность

Все 6 страниц пиксель-в-пиксель с ЗАХАРДКОЖЕННЫМ контентом. Это и есть «дизайн один-в-один» — релизится первым, без CMS.

2

Паспорт + Media

Global SiteSettings + Media-коллекция. Перевод контактов / ссылок / картинок на CMS-слоты.

3

Коллекции контента

Products / Brands / Shops. Catalog и Our Brands — из CMS, ручной ввод карточек.

4

Автосинк на будущее · «закос»

Авто-подтяг товаров по sku-шву из внешнего фида. Posts / новости.

Синтез · кульминация

Один архитектурный ход снимает три напряжения разом

Захардкоженный layout + типизированные CMS-слоты. Layout прибит в React, CMS лишь заполняет размеченные поля. Это развязывает узел, вокруг которого крутилась вся сессия.

1
Дизайн 1:1 не ждёт CMS
Фаза 1 релизится с захардкоженным контентом. Админка подключается слоями позже.
2
Менеджер не ломает вёрстку
Он физически может править только размеченные слоты — структура недоступна.
3
MVP быстрее
Меньше поверхности для ошибок: в CMS только «живое», всё остальное — гвоздями.
Один ход
layout := hardcoded(React) · content := typed_slots(Payload)

Приоритет «дизайн 1:1» и вторичность CMS решаются одним и тем же архитектурным решением. Одно решение — три развязанных напряжения.

1Payload CMS 3.0 внутри Next.js App Router (local API, self-host)
2Захардкоженный layout + типизированные слоты — снимает три напряжения
3Паспорт компании = один Global SiteSettings
4Products с sku-швом готов к будущему автосинку
5Поэтапный роадмап: дизайн 1:1 первым (Ф1), CMS слоями потом (Ф2–4)
