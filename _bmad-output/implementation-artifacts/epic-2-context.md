# Epic 2 Context: Contact-форма и захват лида

<!-- Generated from planning artifacts. Regenerate with compile-epic-context if planning docs change. -->

## Goal

Единственная функция сверх пиксельного визуала в Фазе 1 — рабочий сквозной путь «форма → CRM». B2B-партнёр оставляет заявку, и она гарантированно доходит в CRM Rollun (без хранения на сайте). Реализуется **один** компонент `ContactForm` с **одним** серверным обработчиком, готовый монтироваться на Home, About и Contact в трёх режимах показа (desktop-модалка, инлайн, mobile-навигация на `/contact`). Это климакс основного пользовательского пути (UJ-1) и критерий успеха SM-2; закрывает FR-9 (отправка в CRM), FR-10 (анти-спам) и FR-11 (единый компонент, три режима).

## Stories

- Story 2.1: Единый компонент ContactForm с валидацией
- Story 2.2: Отправка заявки в CRM через Server Action + состояния
- Story 2.3: Анти-спам защита
- Story 2.4: Три режима показа формы

## Requirements & Constraints

- **Успех:** валидная отправка → 2xx от CRM → явное подтверждение (в прототипе кнопка становится `THANK YOU ✓`, затем reset; в модалке — закрытие через ~1.4с).
- **Ошибка:** сбой доставки в CRM → понятное сообщение об ошибке + сохранение всех введённых данных (включая выбранную тему) для повтора; провал POST логируется server-side. Данные пользователя не теряются.
- **Валидация** на единственном пути: обязательные поля заполнены, корректный формат email, тема строго из фикс-списка. Клиентская подсказка дублируется серверной валидацией.
- **Анти-спам:** honeypot-поле + серверная валидация; не прошедшие отправки не уходят в CRM; живой пользователь проходит без дополнительного трения.
- **Не хранить лид на сайте:** нет коллекции `Submissions`, нет mailer — лид только транзитом через Server Action в CRM.
- **Безопасность:** URL и секрет CRM браузеру не видны (server-side POST).
- Контент англоязычный; микрокопи фиксирована («гвоздями»), меняется только в коде.
- Открытые допущения (не блокируют начало, но блокируют прод-включение): контракт CRM API (поля тела / формат / авторизация) — от владельца, до этого endpoint через env `CRM_API_URL`; captcha (провайдер reCAPTCHA/hCaptcha/Turnstile TBD) добавляется поверх honeypot при необходимости.

## Technical Decisions

- **Единый компонент + единый обработчик (AD-8):** ровно один `ContactForm` и одна Server Action. Submit идёт `островок → Server Action → внешний CRM`, минуя любое хранилище контента. Server Action имеет **фиксированный контракт результата** (единый тип ответа success/error), который потребляют все три режима; режимы различаются **только презентацией**, не путём отправки и не набором полей.
- **Расположение:** компонент и его Server Action живут в `components/contact-form/` (обособлено от `components/islands/`). `ContactForm` — client-островок во всех режимах; контент получает пропсами, сам не фетчит.
- **Поля (фиксированы, в этом порядке):** `Your Name`, `Your Email`, `Phone Number`, `Company`, `Question Topic` (select), `Your Message`. Кнопка сабмита — `ASK A QUESTION`.
- **Список тем select (фикс-порядок):** `Select a topic` (disabled) · Wholesale & distribution · Partnership · Marketplace operations · Returns & support · Other.
- **Env-конфиг:** адрес CRM — через `CRM_API_URL` (UPPER_SNAKE, не хардкод).
- Дизайн — источник истины (AD-13); визуальный пиксель приоритетнее a11y при конфликте.

## UX & Interaction Patterns

**Три режима показа (один компонент, один endpoint):**

1. **Desktop-модалка** (Home, About) по триггеру `GET IN TOUCH`: двухколоночная — слева форма, справа тёмная инфо-панель (`.cf-info`: адреса, телефоны, email, соц-иконки GitHub/LinkedIn). `role="dialog"`; закрытие крестик / backdrop-клик / Esc; `body.overflow:hidden` (scroll-lock).
2. **Инлайн** (Contact, `#contactForm`): форма в потоке страницы, без оверлея. Deep-link `?topic=` предвыбирает тему (нормализация текста опции), делает smooth-scroll к форме, кратковременно подсвечивает `.cf-prefilled` и ставит автофокус на `Your Name`.
3. **Mobile Home/About:** триггер не открывает модалку, а навигирует на `/contact` (по возможности с пробросом `?topic=`). На mobile About контакт-модалки нет вовсе.

**Дизайн-токены поля (`.cf-field`):** label Poppins 14px/500, `margin-bottom:8px`; поле input/select/textarea — Roboto 15px, `padding:13px 15px`, `border:1px solid #cfcfcf`, радиус 6px, белый фон; focus → бордер оранж + `box-shadow:0 0 0 3px rgba(239,127,26,0.15)`; textarea `min-height:150px`, `resize:vertical`. Предзаполненный select (`?topic=`) — оранжевый бордер + пульс-анимация. Сетка `.cf-row` двухколоночная, схлопывается в одну колонку.

## Cross-Story Dependencies

- **Story 2.1 → 2.2/2.4:** структура полей и контракт формы из 2.1 используются обработчиком (2.2) и всеми режимами показа (2.4).
- **Story 2.3** встраивает honeypot в тот же единственный путь submit из 2.2.
- **DS-токены и шрифты** (`.cf-field`, focus-ring, Poppins/Roboto) поставляет Epic 1 (Story 1.2) — форма опирается на уже существующие токены.
- **Downstream:** Home (Epic 3) и About (Epic 4) монтируют форму в режиме desktop-модалки / mobile-навигации; Contact (Epic 6, Story 6.3) монтирует её инлайн. Эти эпики зависят от Epic 2, но реализуются после — Epic 2 standalone доводит путь «форма → CRM» до готовности.
- **Прод-включение** блокируется внешним допущением: контракт CRM API от владельца (до этого — env-endpoint).
