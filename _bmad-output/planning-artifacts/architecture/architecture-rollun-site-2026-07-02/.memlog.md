- (version) Next.js 16.2.x — текущий стабильный (16 вышел окт 2025, 16.2.7+ на июнь 2026). Turbopack по умолчанию (dev+build); Cache Components — явная модель кэширования вместо неявного fetch-кэша; params/cookies/headers async-only; Node 20+. App Router — стабильный рекомендованный путь. Проверено веб июл 2026.
- (version) Payload CMS 3.x — текущий стабильный (3.77 фев 2026, релизы до июня 2026). 4.0 в разработке, НЕ стабилен → биндим 3.x, не '3.0'. Next-native (ставится в /app, local API, без сетевого хопа), Postgres+Lexical помечены stable; есть SQLite/Mongo адаптеры. Проверено веб.
- (version) Tailwind CSS v4.x — текущий. CSS-first: токены в директиве @theme в CSS (нет tailwind.config.js), эмитятся и как :root-переменные, и как утилиты; авто-детект контента. Это ЕДИНЫЙ источник DS-токенов. Проверено веб.
- (version) Карта About: d3@7 + topojson-client@3 + us-atlas@3 (states-10m) — биндим ровно как пинит прототип (пиксель-паритет карты), вендорим локально под self-host. React 19 идёт с Next 16; TypeScript 5.x.
- (question) СУБД для Payload не задана в PRD/addendum. Postgres (stable+рекомендован для self-host, Lexical stable на нём) — дефолт-допущение; владельцу подтвердить (альтернативы SQLite для простоты / Mongo).
- (direction) Парадигма: Islands Architecture поверх захардкоженной оболочки с типизированными слотами контента. Страницы/секции — Server Components (RSC), интерактив вынесен в явные client-островки-листья; контент течёт в одну сторону Payload local API -> RSC -> HTML.
- (decision) AD server/client-граница: дефолт — Server Component; 'use client' только на листовых островках интерактива (drawer/shrink/accordion мобильного шасси, product-line слайдер Home, count-up/reveal IO, D3-карта About, Catalog card-slider+quick-view+marquee-hover-pause, ContactForm во всех режимах, Contact map-табы, Our Brands lightbox mobile). Prevents: страница помечает целую секцию client -> ломает SSR/пиксель/перф; одно поведение реализовано по-разному на разных страницах.
- (decision) AD одна DS через Tailwind v4 @theme: все токены прототипов (цвета/типошкала/spacing/radii/shadow/easing) в одном @theme-блоке (эмит: :root-переменные + утилиты). Desktop и mobile — разные раскладки, но резолвят ОДИН набор токенов; поверхностно-специфичные значения (mobile серый --bg, --shell-w:440px) скоупятся, не дублируют имена. Осиротевший colors_and_type.css (Archivo/#EA7B08) игнор. #EF7F1A — единственный оранж. Prevents: 6 страниц+mobile копируют inline :root -> дрейф оранжа/весов/spacing.
- (decision) AD desktop/mobile раскол: две ОТДЕЛЬНЫЕ композиции на страницу (не один адаптивный tree), переключение на ~768px; desktop держит свои внутренние брейкпоинты 560-1100. [ASSUMPTION механизм: обе композиции в DOM, переключение CSS-медиа на 768px, обе SSR -> нет hydration-mismatch от UA-сниффинга; кандидат на пересмотр если вес DOM критичен]. Prevents: одна страница только-CSS-адаптив, другая два дерева -> mobile-поведения садятся вразнобой, пиксель рушится.
- (decision) AD контент в одну сторону: RSC читают контент через Payload local API (getPayload) на request/build; client-островки получают контент пропсами, НИКОГДА не фетчат сами. Prevents: клиентский фетч контента (waterfalls, не-SSR) на одной странице vs серверный на другой; ослабляет 'менеджер физически не ломает дизайн'.
- (decision) AD типизированные слоты: один Payload Global на страницу (HomeContent/AboutContent/CatalogContent/BrandsContent/ShopsContent/ContactContent) + SiteSettings-паспорт + коллекции (Products/Brands/Shops/Media/Posts/Users). НЕТ абстрактной Page, НЕТ block/page-builder. Слот = типизированное поле под маркер захардкоженной вёрстки. Prevents: одна страница как гибкие блоки, другая как фикс-поля -> инвариант 'нельзя сломать дизайн' фрагментируется.
- (decision) AD граница текучести (код vs контент) фикс по 4 уровням: живое (картинки/товары)->CMS · правимый текст живых блоков->CMS · паспорт (тел/адрес/соц/часы)->SiteSettings · гвоздями (вёрстка/структура/юр+статик-текст/микрокопи)->код. Prevents: страница A правит заголовок через CMS, страница B хардкодит тот же класс текста -> разная редактируемость + случайная поверхность слома дизайна.
- (decision) AD единый компонент формы -> единый приёмник: один ContactForm + один серверный обработчик POST-ит на CRM_API_URL (env). Три режима: desktop-модалка (Home/About), инлайн (Contact), mobile-триггер -> /contact. Нет Submissions, нет mailer, лид на сайте не хранится. CRM URL/секрет не доходят до браузера (server action/handler-прокси). Анти-спам honeypot+серверная валидация на единственном пути. Prevents: три страницы строят свою форму (разные поля/валидация/endpoint) или локальное хранение лида.
- (decision) AD офферы — рантайм, не поле: offers генерятся при рендере (buildOffers + репрезентативная цена): Health->Amazon/eBay, Automotive->Amazon/eBay/Walmart. Не Payload-поле. Шов sku/externalId зарезервирован на Products с Фазы 1; фид-перезаписываемые поля (цена/картинки/наличие) изолированы под Фазу 4. Prevents: офферы как редактируемые данные (дрейф, неверные маркетплейсы на линию) или ретрофит фид-шва потом.
- (decision) AD ревалидация: правки CMS видны в проде без пересборки через on-demand revalidation (revalidatePath/Tag из Payload afterChange-хуков), согласовано с явной моделью кэша Next 16 (Cache Components). Страницы статик-рендер + ревалидация по изменению контента; без per-page ad-hoc кэша. Фаза 1 (хардкод) полностью статична. Prevents: одна страница вечно закэширована (правки не видны), другая динамична (медленно).
- (decision) AD вендоринг внешних ассетов под self-host: шрифты Poppins/Roboto/Karla/Caveat через next/font (без рантайм Google Fonts); D3/topojson/us-atlas в бандл (без jsDelivr рантайм); react/babel-standalone скаффолдинг прототипа About выкинуть. - (decision) AD фаза-1-в-форме-слота (шов роадмапа): каждая страница — чистая функция типизированного объекта контента; Фаза 1 подаёт статический инстанс, Фазы 2-3 — Payload Global. - (decision) AD один процесс деплоя: один Next.js 16 процесс (Node 20+) отдаёт публичный сайт + Payload-админку (/admin) + local API в одном рантайме; БД — адаптер Payload (Postgres — допущение).
