# Продуктовый бриф: Сайт Rollun

> Назначение документа: **заготовка для PRD и архитектуры** (внутренний рабочий документ). Детальная модель контента, роадмап по фазам и спецификация дизайн-фиделити вынесены в `addendum.md`.

## Executive Summary

Rollun — американская компания (LLC, Sheridan, WY; команда из Украины, с апреля 2015), которая занимается автоматизированной e-commerce дистрибуцией: автозапчасти и мото-аксессуары (масла и жидкости, электрика, шины и диски, АКБ) и health-продукты (ортопедия, обезболивание и восстановление, добавки), плюс собственные IT/AI-решения. Продажи идут через ведущие маркетплейсы и собственные каналы.

Текущий сайт написан на устаревшей технологии и находится в плохом состоянии — он не доносит до партнёров, что Rollun — серьёзная, технологичная компания. Этот проект — **новое направление: полный редизайн сайта**, воссоздающий готовый handoff-дизайн **пиксель-в-пиксель** на современном стеке **Next.js (App Router) + TypeScript + React**, с контентом, который менеджеры правят сами через **Payload CMS 3.0** — без участия разработчика.

Ключевой архитектурный ход, снимающий главное напряжение проекта: **вёрстка захардкожена в React, а CMS заполняет только заранее размеченные типизированные слоты** (hero-картинка, заголовок блока, текст, контакты). Дизайн выходит 1:1 и не ждёт CMS, менеджер физически не может сломать вёрстку, а MVP собирается быстро. Первый релиз — все страницы пиксель-в-пиксель плюс **рабочая форма Contact** (заявки уходят на почту; интеграция с CRM — позже).

## Проблема

- **Старый сайт роняет доверие.** Он на устаревшей технологии и в плохом состоянии. Для B2B-аудитории (партнёры, wholesale, marketplace-операции) сайт — это витрина серьёзности компании; текущий сайт этому противоречит.
- **Контент правит только разработчик.** Поменять картинку на главной, обновить телефон/адрес при переезде, сменить соц-ссылки — сегодня это задача на код. Менеджеры не автономны.
- **Форма обратной связи должна работать, а не только выглядеть.** Партнёрские и wholesale-заявки — это бизнес-функция; «красивая, но неработающая форма» неприемлема.

## Решение

Новый сайт на **Next.js App Router + TypeScript + React**, воссоздающий handoff-дизайн **пиксель-в-пиксель** (матчим визуал макета, а не копируем DOM прототипа). Контентом управляет **Payload CMS 3.0**, работающая **внутри того же Next.js-процесса** (local API, без сетевого хопа, self-host на инфраструктуре Rollun, opensource). React Server Components тянут данные из Payload напрямую; картинки — через `next/image`; правки менеджера появляются быстро через on-demand revalidation / ISR без пересборки.

Граница между «живым» и «прибитым гвоздями»: картинки, товары, тексты блоков и «паспорт компании» (телефон/адрес/соц-ссылки/часы) — редактируемы через CMS; вёрстка, структура, юр/статичный текст и микрокопи — в коде. Форма Contact в первом релизе отправляет письмо на `llc@rollun.com` (простой mailer); маршрутизация заявок в CRM — будущая интеграция.

## Что делает это решение особенным

Одно архитектурное решение — **«захардкоженный layout + типизированные CMS-слоты»** — снимает разом три напряжения, которые обычно конфликтуют: дизайн 1:1 против разработки CMS, свобода менеджера против целостности вёрстки, скорость MVP против гибкости page-builder'а. Преимущество здесь — не технологический «ров», а точность исполнения и правильная граница между кодом и контентом.

## Кого обслуживаем

- **Посетители сайта (первичная аудитория) — B2B-партнёры.** Оптовые покупатели, потенциальные партнёры, marketplace-операторы. Приходят оценить серьёзность компании и выйти на контакт (форма Contact: wholesale, партнёрство, marketplace-операции, возвраты/поддержка). Покупка как таковая уходит на маркетплейсы — сайт не интернет-магазин.
- **Редакторы контента (вторичная аудитория) — роли `admin` / `manager`.** 1-2 нетехнических сотрудника, которые через админку Payload меняют картинки, товары, «паспорт компании» и тексты живых блоков. `[ASSUMPTION: интерфейс админки — англоязычный стандартный Payload; 1-2 человека]`

## Критерии успеха

- **Пиксель-в-пиксель.** Все страницы (6 desktop + mobile) визуально совпадают с handoff-дизайном на целевых брейкпоинтах — это приоритет №1 и главный критерий приёмки.
- **Форма работает.** Отправка формы Contact реально доставляет письмо на `llc@rollun.com` в первом релизе.
- **Автономность менеджера.** Пользователь с ролью admin/manager может сменить картинку, товар и «паспорт-контакты» через админку без участия разработчика; правка «паспорта» в одном месте обновляется во всех местах сайта.
- **Срок.** Фаза 1 (дизайн 1:1 + рабочая форма) готова в срок «пара дней». `[ASSUMPTION: реалистично, т.к. сборку выполняет AI-агент; критерий — не блокеров к деплою]`

## Скоуп

**В первом релизе (Фаза 0–1):**
- Скаффолд Next.js + Payload в одном проекте; порт DS-токенов и шрифтов; базовый layout (шапка/футер).
- Все страницы (Home, About Us, Catalog, Our Shops, Our Brands, Contact) пиксель-в-пиксель, desktop + mobile, с **захардкоженным** контентом.
- **Рабочая форма Contact** — доставка письма на почту (простой mailer) **и** сохранение заявки в Payload-коллекцию `Submissions` (лид не теряется, виден в админке с первого дня; будущая CRM читает из этого шва).

**Позже (Фазы 2–4, см. `addendum.md`):**
- Фаза 2: `SiteSettings` (паспорт компании) + Media-коллекция; перевод контактов/ссылок/картинок на CMS-слоты.
- Фаза 3: коллекции `Products` / `Brands` / `Shops`; Catalog и Our Brands из CMS (ручной ввод карточек).
- Фаза 4: авто-подтяг товаров из внешнего фида по sku-шву; Posts/новости; интеграция формы с CRM (CRM забирает лиды из коллекции `Submissions`).

**Явно вне скоупа сейчас:**
- Интернет-магазин / корзина / оплата (покупка — на маркетплейсах).
- Авто-синхронизация товаров из внешнего источника (модель к ней готова через поле-шов `sku`, но ввод пока ручной).
- Интеграция формы с конкретной CRM-системой `[ASSUMPTION: система TBD]`.
- Миграция контента/данных со старого сайта.

## Ограничения и ключевые решения

- **Стек фиксирован:** Next.js App Router + TypeScript + React + Payload CMS 3.0 (self-host на инфре Rollun, opensource, local API в одном процессе).
- **Дизайн — из handoff-бандла** `rollun_handoff/rollun-web-site/` (HTML-прототипы + DS-токены + шрифты); источник истины по визуалу.
- **Срок:** пара дней на Фазу 1; исполнитель сборки — AI-агент.
- **Приоритет:** при конфликте «дизайн 1:1» побеждает; единственное, что также обязательно в Фазе 1 — рабочая доставка формы.

## Риски и открытые вопросы

- **Сжатый срок vs пиксель-фиделити.** «Пара дней» на 6 страниц × (desktop+mobile) 1:1 — амбициозно; риск срезания углов на mobile-адаптиве. Митигируется тем, что дизайн уже готов и layout захардкожен.
- **Доставка почты.** Нужен рабочий SMTP/почтовый провайдер и анти-спам (валидация, honeypot/captcha) для публичной формы — детали в PRD. `[ASSUMPTION: доступен транзакционный mailer на инфре Rollun]`
- **CRM-интеграция.** Целевая CRM-система не названа — оставляем «шов» в обработчике формы, но не проектируем интеграцию сейчас.
- **Домен/деплой/переключение** со старого сайта на новый — процесс не описан; вынести в PRD/архитектуру.

## Видение

Сайт 1:1 по дизайну в проде; менеджер сам управляет картинками, товарами и «паспортом компании» через админку с авто-оптимизацией изображений. Дальше — товары начинают авто-подтягиваться из внешнего фида по sku-шву (ручной ввод уходит), заявки с формы льются в CRM через интеграцию, появляются новости/posts. Сайт становится живой, самоуправляемой витриной технологичной дистрибуции Rollun — тем самым доказательством «серьёзной компании», ради которого затевался редизайн.
